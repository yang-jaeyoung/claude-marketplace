
## 1. 모호한 요청 → 명확한 명령

**나쁜 예시 (LLM이 이해하기 어려움):**
```
"Python 게임 개발해줘"
```

**문제점:**
- 어떤 종류의 게임인지 불명확
- AI의 응답: 단순한 예제 코드만 제공
- 사용자 반응: "멀티플레이어 게임을 원했는데..."

**좋은 예시 (명확한 프롬프트):**
```
"Python으로 리얼타임 멀티플레이어 카드 게임을 만들어줘. 
라운드 기반이고, 서버-클라이언트 구조를 사용해야 하고, 
최대 4명의 플레이어를 지원해야 해."
```

**개선 포인트:**
- **구체성(Specificity)**: 게임 유형, 플레이어 수, 아키텍처 명시
- **명확성(Clarity)**: 문장이 길지만 각 요구사항이 명확함

**결과**: 이렇게 재작성된 프롬프트에 대한 AI 응답이 원본보다 **70% 이상 더 나음** [arxiv](http://arxiv.org/pdf/2503.16789.pdf)

## 2. 기술 요청의 세부 사항 추가

**약한 프롬프트:**
```
"압축 알고리즘 코드 작성"
```

**문제점:**
- 어떤 압축? (손실? 비손실?)
- 어떤 언어? 어떤 데이터 타입?

**개선된 프롬프트:**
```
"Python으로 텍스트 데이터의 비손실 압축 알고리즘을 구현해줘. 
Huffman 코딩을 사용하고, 입력 문자열을 받아 압축률을 출력하는 
기능도 포함해야 해. 사용 예제도 보여줘."
```

**개선 포인트:**
- **구체성**: 비손실(lossless) 명시, 알고리즘 지정(Huffman)
- **형식 지정**: 출력 형식(압축률), 예제 포함 요청
- **기술 컨텍스트**: 텍스트 데이터 타입 명시

**AI가 파악한 가정:**
- "사용자는 텍스트 데이터의 비손실 압축이 필요하다"
- "교육 목적 + 실제 사용 예제를 원한다"

**결과**: 반구조화된 Huffman 구현 + 사용 예제 제공 [arxiv](http://arxiv.org/pdf/2503.16789.pdf)

## 3. API 설계 관련 조언 요청

**모호한 요청:**
```
"REST API 보안 어떻게 해야 해?"
```

**명확한 재작성:**
```
"ASP.NET Core에서 JWT 기반 REST API를 구축할 때, 
다음을 고려해서 설명해줘:
1. Access Token / Refresh Token 분리 전략
2. LDAP 통합 인증 (우리는 Active Directory 사용)
3. Rate Limiting과 CORS 설정
4. 각 레이어(Controller/Service/Repository)의 권한 검증

Code 예제는 C#로 부탁하고, 실제 프로덕션 환경을 가정해줘."
```

**개선 포인트:**
- **역할 부여**: "ASP.NET Core 백엔드 엔지니어로서"
- **구체적 컨텍스트**: JWT, LDAP, AD 등 구체 도구 명시
- **구조화**: 1번-4번으로 명확하게 순서 지정
- **출력 형식**: C# 코드 예제 요청, 프로덕션 가정

**AI의 인식:** 
- 사용자는 .NET 전문가 수준의 보안 설계가 필요
- 레거시 AD 인프라와의 통합이 중요

## 4. 대화 반복을 통한 정제 (Interactive Refinement)

**첫 번째 요청:**
```
"Vector DB와 LLM 통합 아키텍처 설계해줘"
```

**AI 응답:** 일반적인 RAG 구조 설명 (너무 광범위)

**개선된 두 번째 요청:**
```
"좋아. 더 구체적으로:
1. PostgreSQL + pgvector 사용 가정 (Pinecone 아님)
2. 임베딩 모델은? (OpenAI embedding vs. 오픈소스)
3. 청킹 전략은? (겹침 비율, 청크 크기)
4. 이전 답변에서 'hybrid search' 언급했는데, 
   pgvector와 전문검색(FTS)을 어떻게 조합해?"
```

**개선 포인트:**
- **피드백**: "좋아" → AI에 긍정 신호, 이어서 심화
- **도구 선택 명시**: Pinecone 제외, pgvector 포함
- **이전 답변 참조**: Hybrid search 언급하며 맥락 유지
- **의사결정 기준**: 비용, 성능, 아키텍처 관점

**결과:** AI가 학습하고, 앞의 응답을 필터링해 맞춤형 설계 제시 [arxiv](http://arxiv.org/pdf/2405.05678.pdf)

## 5. 멀티모달 의도 전달

당신이 LLM과 상호작용할 때:

**텍스트만:**
```
"이 에러 메시지 어떻게 고쳐?"
```

**멀티모달 (음성 + 텍스트 + 코드):**
```
[음성]: "좀 이상한데, 여기 에러를 들려줄게"
[텍스트]: "위 에러가 발생하는 스택 트레이스:"
[코드 스니펫]: <정확한 코드 붙여넣기>
[형식 지정]: "테이블로 정렬해서 각 단계 설명해줘"
```

**효과:**
- 음성 톤 전달 → AI가 긴급도 감지
- 정확한 스택 트레이스 → 가정 최소화
- 구조화된 출력 요청 → 답변 품질 향상

## 6. 프롬프트 재작성의 성공 비율 (실제 데이터)

| 도메인 | 의도 | 개선율(Win%) | 설명 |
|--------|-----|------------|------|
| 소프트웨어/웹개발 | 정보 검색 | **71.6%** | 구체성+명확성 높음 |
| 수학/논리 | 정보 검색 | **74.8%** | 단계별 설명 용이 |
| 소프트웨어/웹개발 | 생성 | **62.7%** | 주관성 높아 개선 낮음 |
| 글쓰기 | 생성 | **59.7%** | 스타일/톤 주관적 |

**핵심:** 정보 검색/기술 문제는 70%+ 개선, 창의 작업은 60% 정도 [arxiv](http://arxiv.org/pdf/2503.16789.pdf)

## 7. 체인-오브-소트(Chain-of-Thought) 유도

**기본 요청:**
```
"REST API 응답 시간을 줄이려면?"
```

**Chain-of-Thought 유도:**
```
"REST API 응답 시간을 줄이려면? 
다음 단계로 생각하면서 설명해줘:
1. 현재 병목 지점이 뭘까? (DB? 캐싱? 네트워크?)
2. 각 병목별로 해결 방안은?
3. PostgreSQL, Redis 활용 시 구체적 쿼리/코드는?
4. 모니터링은 어떻게?

각 단계를 개별 섹션으로 나눠서 작성해줘."
```

**효과:**
- AI가 단계별 사고(reasoning) 수행
- 깊이 있는 분석 → 표면적 답변보다 정확도 20-30% 향상 [microsoft](https://www.microsoft.com/insidetrack/blog/five-tips-for-prompting-ai-how-were-communicating-better-at-microsoft-with-microsoft-copilot/)
